1. Check Your Meta Tags (30-Second Sanity Check)

Before blaming the platforms, confirm your tags are actually correct. Open your page, View Source (not DevTools — the DOM can differ from the raw HTML that scrapers read), and verify:

OG Image Meta Tag Checklist

  • ✓ og:image uses property= not name= (Open Graph uses the property attribute; name is for Twitter Cards)
  • ✓ The image URL is absolute, not relative (https://yoursite.com/og.png, not /og.png)
  • ✓ The image URL uses HTTPS — Facebook and LinkedIn will not fetch HTTP images
  • ✓ og:image:width is 1200 and og:image:height is 630 (or at minimum 200x200)
  • ✓ You have both og:image AND twitter:image tags (they are different standards)
  • ✓ twitter:card is set to summary_large_image (not summary, which shows a tiny thumbnail)

Here is the minimal set of tags that covers every platform: 

<!-- Open Graph (Facebook, LinkedIn, Slack, Discord) -->
<meta property="og:title" content="Your Page Title">
<meta property="og:description" content="A compelling description.">
<meta property="og:image" content="https://yoursite.com/og-image.png">
<meta property="og:image:width" content="1200">
<meta property="og:image:height" content="630">

<!-- Twitter Card -->
<meta name="twitter:card" content="summary_large_image">
<meta name="twitter:image" content="https://yoursite.com/og-image.png">
Common Pitfall: og:image vs og:image:url

og:image and og:image:url are not the same thing. og:image:url is an older, deprecated specification that some scrapers (particularly older LinkedIn crawlers) still check first. If you define both and they point to different images, you get unpredictable behavior. Use only og:image.

2. The Cache Trap: Why Your "Fix" Didn't Fix Anything

This is the single most common reason people think their OG image is broken when it is not. You fix the tags, you paste the link again... and the old broken preview still shows up. It is not broken. It is cached.

Every major platform scrapes OG data once and caches the result aggressively. Your fix is invisible because the platform never re-scraped your page.

Here are the cache durations and the tools you use to clear them:

Facebook / Instagram ~30 days Sharing Debugger Paste URL, click "Scrape Again". Do this twice — the first scrape warms the cache, the second confirms the fix.
Twitter / X ~7 days Card Validator Paste URL, click "Preview Card". Note: Twitter's validator has been flaky since 2024. If it errors, post the link in a DM to yourself — that forces a scrape.
LinkedIn ~7 days Post Inspector Paste URL, click "Inspect". LinkedIn is the most aggressive cacher — you may need to wait 5 minutes between scrapes.
Slack ~1 hour No public debugger Append ?v=2 to your URL to force a fresh scrape. Wait 5 minutes between attempts.
Discord ~1 hour No public debugger Same trick: append ?v=2 to your URL. Discord's scraper respects query strings as unique URLs.
WhatsApp / iMessage ~24-48 hours No debugger The least predictable. Wait it out or use a URL shortener to create a "new" URL.
Pro Tip: The Query String Trick

Appending a query parameter like ?v=2 or ?refresh=1 to your URL forces every platform to treat it as a brand new page. This bypasses ALL caches simultaneously. Once the preview works, remove the parameter for normal sharing — or keep it, if your page ignores unknown query params.

One more thing about Facebook: the Sharing Debugger shows you exactly what the scraper sees, including any warnings or errors. It will tell you if your image is too small, too slow to load, or blocked by robots.txt. Always check the debugger output before assuming your tags are wrong.

3. The 5-Second Rule Nobody Documents

Every major platform's scraper has a timeout. If your OG image takes longer than approximately 5 seconds to load, the scraper silently drops it and moves on. No error message. No warning. Your link just shows up without an image.

This is not documented anywhere official. But ask anyone who has debugged OG images at scale and they will tell you: slow images are the silent killer.

Common causes of slow image loads:

  • Huge PNG files — a 2MB OG image on a $5/month shared hosting plan can take 8+ seconds to serve
  • No CDN — serving images from a single-origin server means Facebook's scraper (which is in California) might be pulling from a server in Frankfurt
  • Cold starts on serverless platforms — if your image is generated on-the-fly by a Lambda/Worker function, the first request after inactivity can take seconds
  • Redirect chains — if your image URL goes through 2-3 redirects, each one adds latency

How to test your image load time from the perspective of a scraper: 

# Test raw download time (this is what scrapers experience)
curl -o /dev/null -s -w 'Total: %{time_total}s | Size: %{size_download} bytes\n' "https://yoursite.com/og-image.png"

# If time_total > 3 seconds, you have a problem.
# Aim for under 1 second. Under 500ms is ideal.

Fix it: compress your PNG (aim for under 200KB), put it behind a CDN (Cloudflare's free tier takes 2 minutes to set up), and enable caching headers so repeat scrapes are instant.

4. Platform-by-Platform Debugging Checklist

Each platform has its own quirks. Here is what to check for each one, in order of how often they cause problems:

Facebook / Instagram

  • Minimum image size: 200x200 pixels. Anything smaller is rejected silently.
  • Maximum file size: 8MB, but anything over 1MB will likely time out (see the 5-second rule above).
  • Supported formats: PNG, JPEG, GIF (static only). WebP is NOT fully supported — Facebook's scraper only began partial WebP support in 2024, and it is still unreliable. Use PNG.
  • Robots.txt check: Facebook respects robots.txt. If your image is in a directory blocked by robots.txt, the scraper will not fetch it. Test: open the image URL in an incognito window.
  • Debugger URL: developers.facebook.com/tools/debug/

Twitter / X

  • Card type matters: summary_large_image shows a full-width image. summary shows a tiny square thumbnail. Make sure you are using the right one.
  • Minimum image size for large image card: 300x157 pixels. Below this, Twitter falls back to a summary card regardless of your tag.
  • Aspect ratio: 2:1 is ideal (1200x600). Twitter crops differently than Facebook — a 1200x630 image works but will have thin bars on the sides.
  • Validator note: The Card Validator at cards-dev.twitter.com has been unreliable since 2024. If it shows an error, try posting the link in a DM to yourself — that triggers a live scrape.

LinkedIn

  • Most aggressive caching: LinkedIn caches OG data for up to 7 days and the Post Inspector can take 5+ minutes to refresh. Be patient.
  • Image sizing: LinkedIn displays OG images at 1.91:1 ratio. 1200x627 works slightly better than 1200x630 (LinkedIn has historically cropped the bottom 3 pixels).
  • Post Inspector URL: linkedin.com/post-inspector/

Slack

  • Unfurl requirements: Slack only unfurls links if your workspace admin has not disabled unfurling and the message is not in a thread where unfurling was collapsed.
  • Cache duration: ~1 hour. The ?v=2 query string trick works reliably.
  • Image sizing: Slack crops OG images to 360x189 in the unfurl preview. Text-heavy images become illegible — keep your OG image text large and centered.

Discord

  • Cache: ~1 hour. Same query string trick applies.
  • No debugger: You test by posting the link in a private server. Discord's embed shows up immediately on first post (no cache on first scrape).
  • Image sizing: Discord displays a smaller thumbnail than Facebook. Text that is readable on Facebook may be too small on Discord.

WhatsApp / iMessage

  • Least predictable: These platforms have opaque caching with no public debuggers. WhatsApp can cache for 48+ hours.
  • Minimum image size: WhatsApp requires at least 300x300 to show a preview image. Under that, it is text-only.
  • Best workaround: Use a URL shortener (like bit.ly) to create a "new" URL that has never been scraped. This bypasses the cache.

5. When the Image Itself Is Broken

Sometimes the tags are correct, the cache is cleared, the load time is fine... and the image still does not show. The problem is the image file itself. Here is what to check:

Wrong Format: The WebP Trap

WebP is an excellent format — smaller files, same quality. But Facebook's scraper did not fully support WebP until mid-2024, and some platforms (older Slack clients, certain email clients that generate previews) still do not. If your OG image is a WebP file, convert it to PNG and try again. This single change has fixed more "mysterious" OG image failures than any other.

Wrong Dimensions

The universal standard is 1200x630 pixels (1.91:1 aspect ratio). But the minimum requirements vary:

  • Facebook: 200x200 minimum, recommends 1200x630
  • Twitter summary_large_image: 300x157 minimum
  • LinkedIn: 1200x627 recommended (matches their exact crop)
  • WhatsApp: 300x300 minimum

If your image is below the platform's minimum, it is silently dropped. Always use 1200x630 — it exceeds every platform's minimum and looks good everywhere.

Text Too Close to the Edge

Platforms crop OG images differently on mobile vs desktop, and some (Slack, Discord) display them at much smaller sizes than Facebook. If your title text extends to within 60 pixels of any edge, it will be cut off on at least one platform. Keep a 60px safe margin on all sides for critical text.

Corrupted or Malformed File

Rare but real: a PNG with a corrupted chunk, a JPEG with bad EXIF data, or an image saved with the wrong file extension (a PNG saved as .jpg). Scrapers are less tolerant of malformed files than browsers. If nothing else works, regenerate the image from scratch. Sometimes a clean re-encode is the only fix.
